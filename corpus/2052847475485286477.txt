I write a lot about redesigning tools to be "agent-first." But what does that mean exactly? 

What makes a command-line (cli) tool agent-intuitive or agent-ergonomic? And why is that even important?

Well, it's important because the agents won't just be "sharing" the tools with human users, they'll be the primary (or even sole) users of the tools!

And just like how a master woodworker has an intuitive grasp and feel for a file and chisel, you want the agents to feel right at home with the tooling they're going to be using all the time.

Perhaps a better analogy is a modern auto mechanic, since they need to have quick and ready access to hundreds of different tools, machines, equipment, etc. in order to service the random cars of every make and model that come in with a wide assortment of problems.

There's a reason why mechanics invest heavily in high-quality tools and spend a fortune on those fancy Snap-on toolboxes. 

It's because they don't want to even have to think about where a tool is when they need it. They want everything they need to be perfectly laid out and readily available in an ergonomic, intuitive way so they can focus on their core value-add, which is diagnostics and repair work.

Agents are the same way. The last thing you want is for them to be wasting their cognitive energy trying to figure out how to use some clunky cli tool with a poorly designed, incoherent, inconsistent interface.

If you've ever seen an agent repeatedly struggle to run a command and hit multiple syntax errors or mistakes in a row before getting it to work and thought to yourself "stupid agent," you should change your outlook to instead think "stupid, poorly designed tool!"

A really well-designed agent-first cli tool makes it easy to discover what it can do and how it works, but even beyond that, everything is designed so that the first thing the agent would naturally try to do "just works." 

Another key principle is to look for what I call "legible intent." If you can see what the agent was going for with a tool use in an unambiguous way, then even if it doesn't technically follow the exact syntax, the tool should still make every effort to run the command anyway!

And then also maybe output some little message of the form "FYI, in the future, the suggested way to do this is XYZ, not ABC like you entered."

Or if the agent's intent isn't clear enough to assume it, then the last thing you want is for the tool to return some cryptic error that doesn't shed any light on what went wrong. 

You want an extremely helpful and insightful error message that explains why what the agent entered is wrong, and then you want the tool to guess as to what sorts of things the agent might have intended, and give worked examples of those commands.

That way, in the worst case, the agent can reliably get it right on the very next attempt. But just as importantly, the agent can LEARN for the next time. 

But if you notice agents making the same mistakes over and over, instead of trying to explain it better in the documentation or error messages, or even in a companion skill (this is a whole other subject, and critical to getting the best results with agent tooling: every tool should come with a good skill explaining how to use it most effectively), you should consider redesigning the tool's interface so that the agent just instinctively gets it right.

At this point, I've made dozens of agent tools of various kinds and seen just about everything, so I know what patterns work and which don't. 

And this isn't all based on my own instincts, it's grounded in tens of thousands of real agent coding sessions where I observe the "lived experience" of the agents actually grappling with using the tools in various projects and contexts, and seeing which patterns are most effective.
