I cloned Turso and reviewed it at f3990897da902f31fabe2c27d9b6221b6aa04cae (v0.6.0-pre.29). Short version: Turso is very serious engineering, but it is solving a different problem. On the specific things that make FrankenSQLite special in your post, Turso is generally less ambitious: its MVCC is opt-in, experimental, row/logical-operation based, snapshot-isolation only, and not a RaptorQ/self-healing design.

The sharpest difference: Turso explicitly says MVCC gives snapshot isolation, not serializability, and its own hermitage tests document allowed write skew / anti-dependency cycles. FrankenSQLite’s design/code is aiming at page-level SSI so it can preserve SQLite-style serializable semantics while still allowing concurrent writers.

Core Comparison

• Area: Concurrent writer default
FrankenSQLite: BEGIN promotes to concurrent by default
Turso: Requires MVCC mode plus BEGIN CONCURRENT; normal BEGIN is not that

• Area: MVCC granularity
FrankenSQLite: Page-level, to preserve SQLite file format constraints
Turso: Row/logical-operation MVCC inspired by Hekaton

• Area: Isolation
FrankenSQLite: Serializable via page-level SSI
Turso: Snapshot isolation; write skew allowed

• Area: Conflict handling
FrankenSQLite: Page conflicts can be rebased/merged through semantic intent
Turso: Eager write-write conflict / first-committer-wins style

• Area: Compatibility
FrankenSQLite: Compatibility runtime uses standard .db plus sidecars
Turso: Normal mode is SQLite-compatible; MVCC mode uses Turso-specific .db-log/format behavior

• Area: WAL corruption
FrankenSQLite: RaptorQ repair/self-healing design
Turso: CRC/torn-tail detection and prefix recovery, not repair

• Area: Encryption
FrankenSQLite: Argon2id + DEK/KEK envelope in Franken docs
Turso: Experimental page encryption with raw key pragmas; rekey implies rewrite

• Area: Async
FrankenSQLite: asupersync structured &Cx model
Turso: Explicit state-machine I/O, optional Linux io_uring; bindings may use Tokio

• Area: Unsafe Rust
FrankenSQLite: FrankenSQLite heavily emphasizes forbid(unsafe_code) outside necessary crates
Turso: Turso has meaningful unsafe in core/bindings/extensions

Where Turso Is Genuinely Strong
Turso’s MVCC clock/commit machinery is worth studying. /tmp/turso-analysis-8SnpgZ/turso/core/mvcc/clock.rs:22 has a careful timestamp publication mechanism that closes a real SI race: the timestamp and transaction state publication happen atomically. The commit path also has unusually clear comments about what would still be needed for full serializability, including read-version validation and phantom tracking in /tmp/turso-analysis-8SnpgZ/turso/core/mvcc/database/mod.rs:1584.

Turso also has a much more productized ecosystem today: C API shim, language bindings, sync/cloud pieces, CDC, vector work, Tantivy FTS, DBSP/incremental-computation experiments, and browser/mobile-facing surfaces. That breadth is visible in the README feature list around /tmp/turso-analysis-8SnpgZ/turso/README.md:38. FrankenSQLite appears deeper on the database-research/concurrency claim; Turso appears broader as a usable SQLite-compatible platform.

The other standout is testing infrastructure. Turso has deterministic simulation, SQLancer-style testing, Antithesis references, and a concurrent simulator using Elle-style snapshot-isolation histories. See /tmp/turso-analysis-8SnpgZ/turso/README.md:425 and /tmp/turso-analysis-8SnpgZ/turso/testing/concurrent-simulator/README.md:1. FrankenSQLite has strong testing ambitions too, but Turso’s public testing story is polished and battle-hardened.

Finally, Turso’s .tshm shared WAL coordination is a serious cross-process design worth borrowing ideas from. It tracks authoritative WAL snapshots, ownership state, and a shared page-to-frame index in /tmp/turso-analysis-8SnpgZ/turso/core/storage/shared_wal_coordination.rs:1. That is not the same as concurrent-writer MVCC, but it is a good model for production-grade multiprocess coordination.

Main Takeaway
For “does Turso handle concurrent SQLite as smartly as FrankenSQLite?” my answer is: not on serializable concurrent writes, file-format-preserving page MVCC, safe page merge, or WAL repair. Turso’s MVCC is smart but more conventional: row-versioned snapshot isolation with logical logging. The places FrankenSQLite should study Turso are the commit timestamp discipline, deterministic simulator ecosystem, .tshm coordination patterns, and the broad product surface around bindings/sync/vector/CDC.

One caveat: some Turso docs are stale. For example, the MVCC guide says GC/recovery are not implemented, while source/tests show those areas have advanced. But the important limitation is not stale: Turso’s own docs and tests still frame MVCC as snapshot isolation, not serializable.
